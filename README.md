# Message Gateway

Message broker/gateway for WhatsApp via Evolution API. Consumes messages from RabbitMQ queues, concatenates text messages from the same sender within a configurable time window, transcribes audio messages via OpenAI Whisper, and produces processed messages to another RabbitMQ queue.

## Tech Stack

- Ruby 3.4.3 / Rails 8.1 (API-only)
- SQLite3 (WAL mode) — no external database
- Solid Queue (background jobs)
- Solid Cache
- Bunny (RabbitMQ client)
- OpenAI Whisper (audio transcription via ruby-openai)

## Architecture

```
RabbitMQ (<instance>.messages.upsert)
  |
  v
MessagesConsumer (manual ack, retry via x-retry-count header)
  |  +-> DeadLetterPublisher -> DLQ (after 3 failures)
  v
IncomingMessageJob (Solid Queue, concurrency=1 per sender)
  |
  +-> MessageParser -> ParsedMessage (value object)
  +-> SenderRegistrationService -> Sender (find_or_create)
  +-> MessageAuditJob (async audit record)
  +-> MessageStrategyResolver
       |
       +-- "conversation" -> ConversationStrategy
       |     +-> MessageConcatenationService
       |           +-> ConcatenationBuffer (append + reset timer)
       |           +-> ConcatenationFlushJob (when timer expires)
       |                 +-> ProcessedMessagePublisher -> PROCESSED_MESSAGES_QUEUE
       |
       +-- "audioMessage" -> AudioMessageStrategy
             +-> AudioTranscriptionJob
                   +-> AudioTranscriptionService (OpenAI Whisper)
                   +-> TokenUsage.create!
                   +-> ProcessedMessagePublisher -> PROCESSED_MESSAGES_QUEUE
```

## Setup

### Prerequisites

- Ruby 3.4.3 (managed via asdf)
- Access to a RabbitMQ instance

### Installation

```bash
git clone <repo-url>
cd message_gateway
cp .env.example .env  # Edit with your RabbitMQ and OpenAI credentials
bin/setup
```

`bin/setup` installs dependencies, prepares the database, and installs git hooks.

### Running Locally

In separate terminals:

```bash
# Terminal 1: RabbitMQ consumer
bin/consumer

# Terminal 2: Background jobs (Solid Queue)
bin/jobs
```

The consumer connects to the RabbitMQ instance defined in `RABBITMQ_URL` and starts processing messages.

## Environment Variables

| Variable | Description | Default |
|---|---|---|
| `RABBITMQ_URL` | AMQP connection URL | `amqp://guest:guest@localhost:5672` |
| `RABBITMQ_QUEUES` | Comma-separated incoming queue names to consume | _(required)_ |
| `PROCESSED_MESSAGES_QUEUE` | Queue name for processed output messages | _(required)_ |
| `OUTGOING_MESSAGES_QUEUE` | Queue name for LLM replies to deliver via WhatsApp | _(optional)_ |
| `MESSAGE_CONVERSATION_CONCAT_WINDOW` | Seconds to wait for more messages before flushing | `30` |
| `MESSAGE_MAX_RETRY_COUNT` | Max retries before sending to DLQ | `3` |
| `OUTGOING_TYPING_DELAY_MS_PER_CHAR` | Typing delay per character in milliseconds | `35` |
| `OPENAI_API_KEY` | OpenAI API key for Whisper transcription | _(required)_ |
| `OPENAI_TRANSCRIPTION_MODEL` | Whisper model name | `whisper-1` |
| `OPENAI_TRANSCRIPTION_LANGUAGE` | Language hint for transcription | `pt` |
| `EVOLUTION_API_URL` | Evolution API base URL (outgoing messages) | _(required for outgoing)_ |
| `EVOLUTION_API_KEY` | Evolution API key (outgoing messages) | _(required for outgoing)_ |
| `SECRET_KEY_BASE` | Rails secret key | _(required in production)_ |

## Processed Message Format

Messages published to `PROCESSED_MESSAGES_QUEUE`:

```json
{
  "id": "sender-uuid",
  "phone_number": "5511999999999",
  "text": "concatenated or transcribed message",
  "name": "Sender Name"
}
```

## Docker (Production)

```bash
docker compose up --build
```

Two containers:
- **consumer** — RabbitMQ message consumer (`bin/consumer`)
- **worker** — Solid Queue background job processor

Both share a SQLite volume. RabbitMQ is external (configured via `RABBITMQ_URL` in `.env`).

## Key Design Decisions

- **Concatenation with sliding window**: Each text message resets the sender's timer. A delayed Solid Queue job checks if the timer truly expired; stale jobs are no-ops.
- **Strategy pattern**: `ConversationStrategy` and `AudioMessageStrategy` are resolved by message type, keeping the orchestrator job clean.
- **Consumer-side dead letter**: Since quorum queues don't have DLX configured, retries are tracked via `x-retry-count` message headers. After 3 failures, messages go to `<queue>.dlq`.
- **UUID primary keys**: All tables use UUID string IDs generated by `SecureRandom.uuid`.
- **Separate SQLite databases**: Primary, queue, and cache databases are separate to reduce write contention.
